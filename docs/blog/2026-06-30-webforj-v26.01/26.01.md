webforJ `26.01` is here, and it puts the JavaScript and Typescript ecosystem one annotation away from a Java class. A new Maven and Gradle build plugin pulls a [Bun](https://bun.sh/)-powered bundler into the build you already run, so a webforJ view can declare an npm package, pull in a [React](https://react.dev/), [Svelte](https://svelte.dev/), [Lit](https://lit.dev/), or other web component, and author [TypeScript](https://www.typescriptlang.org/), [SCSS](https://sass-lang.com/), [Less](https://lesscss.org/), or [Tailwind](https://tailwindcss.com/) — all without a Node toolchain, a separate frontend project, or a second build to run.

The development watch that comes with it has been reworked end to end, with unified live reload across Maven Jetty, Spring Boot, and JRebel. Alongside the new build paradigm, `26.01` ships a brand-new `Upload` component, browser APIs for geolocation, page visibility, and app icon badges, and pinning and search inside `AppNav`. As always, the [GitHub release overview](https://github.com/webforj/webforj/releases/tag/26.01) has the complete picture.

## A new way to build: the webforJ build plugin

The headline change in `26.01` is the **webforJ build plugin** — a Maven and Gradle plugin that hands the JavaScript ecosystem to a Java developer through a single annotation. Add it once:

```xml title="pom.xml"
<plugin>
  <groupId>com.webforj</groupId>
  <artifactId>webforj-maven-plugin</artifactId>
  <extensions>true</extensions>
</plugin>
```

New webforJ archetypes ship with it preconfigured; existing projects opt in incrementally.

### Bun-powered frontend bundler

Under the hood, [Bun](https://bun.sh/) does the heavy lifting, downloaded and cached for you. Take React. A view names the packages it needs, points at a TSX entry, and uses the result like any other element:

```java title="ReactView.java"
@Route("/react")
@BundlePackage(value = "react", version = "^19.0.0")
@BundlePackage(value = "react-dom", version = "^19.0.0")
@BundleEntry("react/counter.tsx")
public class ReactView extends Composite<FlexLayout> {
  public ReactView() {
    getBoundComponent().add(new Element("react-counter"));
  }
}
```

The TSX entry exposes a React component as a custom element. No adapter library:

```tsx title="src/main/frontend/react/counter.tsx"
import React, { useState } from 'react';
import { createRoot } from 'react-dom/client';

function Counter() {
  const [count, setCount] = useState(0);
  return <button onClick={() => setCount(c => c + 1)}>clicked {count} times</button>;
}

class ReactCounter extends HTMLElement {
  connectedCallback() {
    createRoot(this).render(<Counter />);
  }
}

customElements.define('react-counter', ReactCounter);
```

Bun installs the packages, compiles the TSX, and loads the result the first time `/react` renders. The same pattern works for [Lit](https://lit.dev/), [Svelte](https://svelte.dev/), or any library that registers a custom element — `new Element("...")` is the universal bridge.

For styling, the bundler treats each compiler as an **extension**. SCSS and Less turn themselves on the moment you write a `.scss` or `.less` source. Tailwind is one toggle away, and once on, it scans your **Java sources** for utility class names and generates a stylesheet with only what you use. Write `card.addClassName("flex", "items-center", "gap-8")` in Java, save, and the utilities are in your page.

:::info Using the bundler
Going forward, the bundler is the default path for new webforJ projects, and our docs and samples will lean on it over the older annotation-based approaches. Your existing `@StyleSheet` and `@JavaScript` annotations keep working — adoption is a per-view decision, not a migration.
:::

### A development watch worth running

The plugin's `watch` goal powers a development loop that's been reworked end to end. Run it alongside your server, and from there the loop feels the same on Spring Boot, Maven Jetty, or JRebel:

- **Stylesheet and image edits patch in place** — no reload, form data and scroll position intact.
- **Script edits reload the view**, because new behavior needs a fresh page.
- **Java edits restart the server, with styles staying applied across the restart.**
- **The browser waits until the app is ready before reloading**, with a small indicator showing while the restart happens.

:::info Learn More
See the [webforJ build plugin](/docs/configuration/build-plugin), [Frontend bundler](/docs/managing-resources/bundler/overview), and [Frontend watch](/docs/configuration/deploy-reload/frontend-watch) docs for setup, options, and live-reload keys.
:::

## `Upload` component

Out of the build story and into the UI. `26.01` opens its component lineup with the **`Upload`** component, an inline file picker that drops into a layout next to the rest of your form. File pickers don't always belong in modals, and this one doesn't have to.

<!-- TODO: ComponentDemo to be added once the verified sample is in webforj-samples. -->

```java
Upload upload = new Upload();
upload.addFilter("Images", "*.png;*.jpg");
upload.addFilter("Documents", "*.pdf;*.docx");
upload.setSelectionMode(Upload.SelectionMode.MULTIPLE);
layout.add(upload);
```

You get drag-and-drop, named filters, single-or-many selection, and directory uploads out of the box. On browsers that support the [File System Access API](https://developer.mozilla.org/en-US/docs/Web/API/File_System_Access_API), users get the native OS picker.

:::info Learn More
See the [`Upload`](/docs/components/upload) documentation for filters, selection modes, drop-zone behavior, and event tracking.
:::

## App icon and favicon badges

Past components, three new browser APIs round out the release. First, a small change with a big footprint: two complementary APIs for painting **notification badges** onto the surfaces users glance at — the OS app icon and the browser tab favicon.

```java
App.setBadge(5);                      // OS app icon (dock, taskbar, home screen)
Page.getCurrent().setIconBadge(5);    // Browser tab favicon
```

`App.setBadge` lights up the macOS dock, the Windows taskbar, and the Android home screen once the app is installed. `Page.setIconBadge` paints the same count onto the favicon — no installation required. Call both and you're covered everywhere a user might be looking. Color, shape, and size are all configurable via `IconBadgeOptions`.

:::info Learn More
See the [App badges](/docs/advanced/app-badges) documentation for prerequisites, browser support, and styling.
:::

## Page Visibility API

Badges land harder when paired with the second new browser API: **Page Visibility**. It tells your code when the user has switched tabs, minimized the window, or come back — so you can pause polling when nobody's looking, escalate to a desktop notification when the tab is hidden, or refresh stale data the moment focus returns.

```java
Page.getCurrent().onVisibilityChange(event -> {
  if (event.isHidden()) {
    pauseRendering();
  } else {
    resumeRendering();
  }
});
```

:::info Learn More
See the [Page Visibility](/docs/advanced/page-visibility) documentation for the full API and a worked example that pairs it with desktop notifications and favicon badges.
:::

## Geolocation API

Rounding out the trio, the browser's geolocation subsystem gets a first-class Java API. Ask for a one-shot position, or register a watch listener and stream updates as the device moves:

```java
PendingResult<GeolocationPosition> request =
    Geolocation.getCurrent().getCurrentPosition();

request.thenAccept(position -> {
  double lat = position.getLatitude();
  double lng = position.getLongitude();
  double accuracy = position.getAccuracy();
});
```

High-accuracy mode, timeouts, and a max-age cache are there when you need them. Failures arrive as a typed `WebforjGeolocationException`.

:::info Learn More
See the [Geolocation](/docs/advanced/geolocation) documentation for the full API, configuration, and a complete example.
:::

## `AppNav` pinning and search

Back to components for one more upgrade. The `AppNav` component picks up two features that scale it to deeper menus: **pinning** lifts a user's most-used items into a group at the top of the nav, and **search** filters the menu as they type.

```java
AppNav nav = new AppNav("main-nav");

nav.getPinning().setEnabled(true);
nav.getPinning().setAutosave(true);   // persists to browser local storage

nav.getSearch().setFieldVisible(true);
nav.getSearch().setPlaceholder("Search");
```

The defaults feel right out of the box: the pin toggle appears on hover and keyboard focus, unpinning sends an item back to its exact original position, and pins survive a reload with zero server code. Search opens groups containing a match and keeps pinned shortcuts visible while filtering.

:::info Learn More
See the [`AppNav`](/docs/components/appnav#pinning) documentation for pin events, custom persistence, and driving search from your own input.
:::

## Nested bean binding

On the data side, a small but well-loved upgrade for data-binding apps. A binding property can now be a **dotted path** into a nested bean:

```java
BindingContext<Person> context = new BindingContext<>(Person.class);
context.bind(streetField, "address.street").add();
context.bind(cityField, "address.city").add();
```

Reads resolve safely through `null` intermediates, writes create any missing intermediate bean automatically, and Jakarta validation annotations are detected the same way as on top-level properties.

:::info Learn More
See the [Bindings](/docs/data-binding/bindings#nested-bean-properties) documentation for the full rules around reads, writes, and validation on nested paths.
:::

## Application Security

Two big additions land in the docs alongside the code. First, a brand-new **[Application Security](/docs/security/application-security/overview)** section. webforJ's [server-driven model](/docs/architecture/client-server) closes off whole classes of web attacks before you write a line of code — but it doesn't remove your responsibilities, and the new section is an honest line drawn between what the framework handles and what stays on you.

- **[Common Threats](/docs/security/application-security/common-threats)** — how XSS, CSRF, SQL injection, and `executeJs` injection look in a webforJ app.
- **[Production Hardening](/docs/security/application-security/production-hardening)** — TLS, server-side validation, secure-by-default routes, dependency upkeep.
- **[Managing Secrets](/docs/security/application-security/managing-secrets)** — keeping passwords, API keys, and tokens out of your repo, with Spring Boot patterns included.
